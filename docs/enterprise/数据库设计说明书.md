# 数据库设计说明书

## 1. 设计目标

当前项目数据库用于支撑 Agent 底座的会话、任务、资产、工具结果和系统治理能力。

当前设计目标：
- 满足阶段 2 的系统治理与业务执行闭环。
- 统一表命名、审计字段和索引规范。
- 为后续 PostgreSQL、分区、分库分表与可视化 trace 演进预留结构基础。

当前数据库选型：
- SQLite

## 2. 设计原则

### 2.1 命名规范

- 表名统一使用 `xxx_xxx` 格式。
- 系统表、授权认证表统一使用 `sys_` 前缀。
- 业务执行表统一使用 `biz_` 前缀。

### 2.2 约束规范

- 所有表必须有主键。
- 当前数据库层不使用外键。
- 表之间的关联关系由应用层和索引策略维护。

这样设计的原因：
- 企业级底座后续很可能进入分区、分库分表与异构存储场景。
- 外键在多库、多租户和异步链路里容易成为迁移与写入瓶颈。
- Agent 底座更重视高可扩展性、可迁移性和链路治理，而不是单机强约束。

### 2.3 审计与扩展字段规范

所有表统一包含以下 5 个审计扩展字段：
- `created_by`
- `updated_by`
- `created_at`
- `updated_at`
- `ext_data1`
- `ext_data2`
- `ext_data3`
- `ext_data4`
- `ext_data5`

说明：
- `created_by`：记录创建主体。
- `updated_by`：记录最后修改主体。
- `created_at`：记录创建时间。
- `updated_at`：记录最后修改时间。
- `ext_data1` 到 `ext_data5`：预留扩展字段，用于兼容后续扩展而不频繁改表。

## 3. 表分类

### 3.1 系统表

- `sys_schema_version`
- `sys_user`
- `sys_request_trace`
- `sys_route_policy`
- `sys_auth_token`
- `sys_agent_registry`

说明：
- `sys_schema_version` 用于记录数据库结构版本，是后续迁移和发布的基线表。
- `sys_user` 归入系统表，因为它承担的是授权认证主体与系统归属角色。
- `sys_request_trace` 归入系统表，因为它承载 trace、鉴权、限流、幂等和错误治理信息。
- `sys_route_policy` 用于承载请求路由规则和策略配置。
- `sys_auth_token` 用于承载 API Key、Token 或后续认证凭据元数据。
- `sys_agent_registry` 用于承载多 Agent 角色注册与能力声明。

### 3.2 业务表

- `biz_session`
- `biz_message`
- `biz_asset`
- `biz_task`
- `biz_tool_result`
- `biz_route_decision`
- `biz_agent_run`

说明：
- 会话、消息、资产、任务、工具执行明细都属于业务执行数据，应和系统治理数据分离。
- 路由决策与 Agent 执行片段属于业务执行过程数据，应纳入业务表管理。

## 5. 关系说明

当前数据库不使用外键，但仍保持如下逻辑关系：
- `sys_user` 1 -> N `biz_session`
- `biz_session` 1 -> N `biz_message`
- `biz_session` 1 -> N `biz_asset`
- `biz_session` 1 -> N `biz_task`
- `biz_task` 1 -> N `biz_tool_result`
- `biz_task` 1 -> N `biz_route_decision`
- `biz_task` 1 -> N `biz_agent_run`
- `sys_request_trace` 通过 `session_id / turn_id / task_id` 关联业务链路
- `sys_route_policy` 与 `biz_route_decision` 通过路由名称或策略标识关联
- `sys_agent_registry` 与 `biz_agent_run` 通过 `agent_id` 关联

## 6. 索引设计

当前已落地索引：
- `sys_schema_version(updated_at DESC)`
- `biz_session(user_id, updated_at DESC)`
- `biz_session(updated_at DESC)`
- `biz_message(session_id, created_at ASC)`
- `biz_message(trace_id)`
- `biz_asset(session_id, created_at ASC)`
- `biz_asset(kind, created_at DESC)`
- `biz_task(session_id, status, updated_at DESC)`
- `biz_task(trace_id)`
- `biz_task(created_at DESC)`
- `biz_task(task_mode, queue_status, updated_at DESC)`
- `biz_tool_result(task_id, created_at ASC)`
- `biz_tool_result(trace_id)`
- `sys_request_trace(task_id)`
- `sys_request_trace(path, status_code, updated_at DESC)`
- `biz_route_decision(task_id, created_at ASC)`
- `biz_route_decision(trace_id)`
- `biz_agent_run(task_id, created_at ASC)`
- `biz_agent_run(trace_id)`
- `sys_route_policy(enabled, priority DESC)`
- `sys_auth_token(subject_id, status, updated_at DESC)`
- `sys_agent_registry(agent_role, enabled, updated_at DESC)`

索引设计原则：
- 列表查询按更新时间或创建时间排序。
- 排障查询优先支持 `trace_id / task_id / session_id`。
- 多模态资产查询优先支持 `kind + created_at`。
- 异步任务查询优先支持 `task_mode + queue_status + updated_at`。
- 请求路由与多 Agent 执行查询优先支持 `task_id / trace_id`。

## 7. 分区、排序、分库分表规划

### 7.1 分区规划

当前 SQLite 不支持真正的分区表，因此当前阶段以“可迁移命名 + 时间索引”代替。

后续 PostgreSQL/MySQL 方案建议：
- `sys_request_trace` 按月分区。
- `biz_message` 按会话活跃时间或创建月份分区。
- `biz_asset` 按创建月份分区。
- `biz_task` 按创建月份分区。
- `biz_tool_result` 按创建月份分区。

### 7.2 排序规划

统一排序建议：
- 列表页默认按 `updated_at DESC` 或 `created_at DESC`
- 会话内消息按 `created_at ASC`
- 工具执行明细按 `created_at ASC`

### 7.3 分库分表规划

后续进入高并发和大数据量阶段时建议：
- 系统表与业务表分库。
- Trace 与业务执行分库。
- 会话、消息、任务、工具结果可按租户或用户维度分片。
- `trace_id / task_id / session_id` 作为跨表、跨库关联主线。

## 8. 字段默认值与约束建议

当前统一默认值策略：
- 主键字段：由应用层生成，不依赖数据库自增。
- 审计字段：
  - `created_by` / `updated_by`：默认由应用层写入，迁移场景可回填 `system_migration`
  - `created_at` / `updated_at`：默认由应用层写入 ISO 8601 时间
- 扩展字段：
  - `ext_data1` 到 `ext_data5` 默认空字符串

当前建议的字典约束：
- `biz_message.role`：`user / assistant`
- `biz_task.status`：`completed / failed / running / pending`
- `sys_request_trace.auth_type`：`api_key / bearer / anonymous`
- `biz_task.task_mode`：`sync / async`
- `biz_task.queue_status`：`pending / queued / running / succeeded / failed / cancelled`
- `biz_agent_run.run_status`：`pending / running / succeeded / failed`

当前 SQLite 未强制 CHECK 约束，原因是：
- 阶段 2 仍在快速演进，应用层枚举变化频率高于数据库结构变化。
- 后续迁移到 PostgreSQL 时更适合统一评估枚举约束和 domain type。

## 9. PostgreSQL 落地预案

后续切换 PostgreSQL 时建议：
- 保持现有 `sys_` / `biz_` 表命名不变。
- 保持无外键原则不变。
- 将时间字段统一切为 `TIMESTAMPTZ`。
- 将大文本字段按实际查询模式评估是否拆表或冷热分层。
- 将 `sys_request_trace`、`biz_message`、`biz_task`、`biz_tool_result` 按月分区。
- 将 `trace_id / task_id / session_id` 作为主要二级索引键。
- 补充正式迁移工具链，例如 Alembic / Flyway 风格版本管理。

建议 DDL 演进顺序：
1. 先建立 `sys_schema_version`
2. 再迁移系统表
3. 再迁移业务表
4. 最后补索引和分区

## 10. 结论

当前数据库设计已经统一到企业级底座规范：
- 有主键
- 无外键
- 有系统表与业务表分层
- 有统一审计扩展字段
- 有数据库版本表
- 有索引策略
- 有分区、排序、分库分表的后续演进规划

后续新增表时必须继续遵守本说明书中的命名、审计和索引规则。

## 11. 面向四项能力的数据库预留结论

数据库设计已经为以下能力预留正式结构方向：
- 请求路由中台
- 异步任务体系
- 权限控制
- 多 Agent 编排
