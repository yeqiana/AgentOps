# 开发计划

## 1. 阶段状态

阶段 1：已完成

## 2. 阶段 2 建议目标

- 先补企业级治理底座
- 引入 `router_node`
- 引入 `review_node`
- 引入多 Agent 编排
- 增强上传后二次分析结果结构化
- 引入后台管理视图
- 引入统一鉴权基础能力
- 引入失败重试与恢复策略
- 引入 trace service 基础能力
- 引入配置中心与版本治理基础能力
- 引入安全边界与工具调用策略
- 引入基础限流、幂等与配额控制

## 3. 阶段 2 开发顺序

1. 设计统一鉴权、限流、幂等方案
2. 设计失败重试、超时、熔断与降级策略
3. 设计 trace service 与观测数据模型
4. 设计配置中心、Prompt 版本与 API 版本治理
5. 设计工具白名单、文件安全与策略控制
6. 在治理底座稳定后，再引入 router / review 状态协议
7. 新增 workflow 节点与多 Agent 编排
8. 补齐 API、数据库与 trace 扩展
9. 增加测试
10. 文档同步

## 4. 阶段 2 新增治理能力

### 4.1 统一鉴权

- API Key / Bearer Token 接入
- 用户身份识别
- 入口级权限校验

### 4.2 失败重试与恢复

- 模型调用重试策略
- 工具调用重试策略
- 超时、退避和失败分类
- 熔断、降级与恢复策略
- 幂等与重复提交保护

### 4.3 Trace Service

- 统一 trace 记录模型
- trace 与 task / tool_results 关联
- 为后续可视化 trace 页面预留接口

### 4.4 配置与版本治理

- 环境配置分层
- Prompt 版本管理
- API 版本管理
- Provider 配置管理

### 4.5 安全边界与策略控制

- 工具白名单
- 文件大小和类型限制
- 敏感信息脱敏
- 审计字段补充

## 5. 阶段 3 建议新增项

- 可视化 trace / 观测页面
- 监控与告警面板
- 更完整的权限治理与多用户策略
- 成本统计与预算控制
- 异步任务调度平台
- 模型路由与策略引擎
- 评测、回归与效果基线平台

## 6. 风险

- 范围蔓延
- 工作流复杂度快速上升
- 数据模型耦合增加
- 治理能力滞后导致阶段 2 返工
- 多 Agent 先于鉴权和限流落地会放大稳定性风险
