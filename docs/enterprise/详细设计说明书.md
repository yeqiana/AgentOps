# 详细设计说明书

## 1. 文档目标

本文档描述 `AgentOps` 当前真实代码结构、核心执行链路、关键模块职责和阶段 2 新增能力的落地方式，重点覆盖：
- 五层结构
- 请求路由中台最小版
- 流式对话
- 异步任务深化
- 治理与排障能力

## 3. 五层结构设计

### 3.1 presentation

这是什么：
- 对外入口层。

做什么：
- 提供 CLI、HTTP API、SSE 接口。
- 处理请求参数、响应模型、中间件和错误转换。

为什么这么做：
- 接入协议变化频繁，必须与应用编排和底层实现隔离。

当前重点目录：
- `app/presentation/cli.py`
- `app/presentation/api/app.py`
- `app/presentation/api/schemas.py`
- `app/presentation/api/middleware/`

### 3.2 application

这是什么：
- 用例编排层。

做什么：
- 组织会话、任务、配置、告警、角色、异步任务等应用服务。
- 负责把接入层请求转成可执行状态。

为什么这么做：
- 业务编排不应直接写在 API 层，也不应和基础设施细节耦合。

当前重点目录：
- `app/application/services/session_service.py`
- `app/application/services/task_service.py`
- `app/application/services/request_route_service.py`
- `app/application/services/async_task_service.py`
- `app/application/services/config_service.py`

### 3.3 domain

这是什么：
- 稳定的核心模型层。

做什么：
- 定义 `AgentState`、任务记录、路由决策记录、trace 记录、RBAC 记录等共享模型。

为什么这么做：
- API、工作流、数据库和工具层都要围绕同一批稳定模型协作。

当前重点目录：
- `app/domain/models.py`
- `app/domain/errors.py`

### 3.4 workflow

这是什么：
- Agent 工作流编排层。

做什么：
- 定义节点、执行协议、角色协议和 LangGraph 图。

为什么这么做：
- Agent 编排逻辑必须独立，后续才能平滑演进到多图、多角色和多 Agent。

当前重点目录：
- `app/workflow/graph.py`
- `app/workflow/nodes.py`
- `app/workflow/registry.py`
- `app/workflow/policies.py`

### 3.5 infrastructure

这是什么：
- 外部依赖与技术实现层。

做什么：
- 提供数据库、LLM、工具、鉴权、trace、告警、队列、存储等能力。

为什么这么做：
- 基础设施最容易变，必须和应用层、工作流层隔离。

当前重点目录：
- `app/infrastructure/persistence/`
- `app/infrastructure/llm/`
- `app/infrastructure/tools/`
- `app/infrastructure/auth/`
- `app/infrastructure/trace/`
- `app/infrastructure/alert/`
- `app/infrastructure/queue/`

## 4. 核心执行链路

### 4.1 同步对话链路

1. API 或 CLI 接收用户输入
2. `TaskService` 生成本轮 `AgentState`
3. `RequestRouteService` 生成入口层路由决策
4. LangGraph 执行：
   - `tool_node`
   - `router_node`
   - `protocol_node`
   - `plan_node`
   - `debate_node`
   - `arbitration_node`
   - `answer_node`
   - `critic_node`
   - `review_node`
5. `SessionService` 持久化消息、资产、任务、工具结果、路由决策
6. `TraceService` 绑定任务上下文
7. 接口返回同步响应或流式响应

### 4.2 流式对话链路

1. 客户端调用 `POST /chat/stream`
2. API 层创建本轮状态
3. `ChatService.stream_turn_events(...)` 按事件输出：
   - `metadata`
   - `answer_delta`
   - `done`
   - `error`
4. SSE 持续回传给调用方
5. 最终结果仍按正常链路持久化

### 4.3 异步任务深化链路

1. 客户端调用 `POST /tasks/submit`
2. API 层校验 `task.submit` 权限
3. `TaskService` 生成本轮状态
4. `AsyncTaskService.persist_queued_turn(...)`
5. `BackgroundTaskRunner` 把任务交给本地线程池
6. 后台线程执行：
   - `queued`
   - `running`
   - `completed / failed`
7. `SessionService` 为任务写入 `biz_task_event`
8. 调用方通过以下接口回查：
   - `GET /tasks/{task_id}`
   - `GET /tasks/{task_id}/events`
   - `GET /tasks/runtime`

## 5. 请求路由中台最小版

这是什么：
- 二阶段新增的入口层请求路由服务。

做什么：
- 在进入工作流前，根据输入、资产类型、上下文和工具情况生成统一路由决策。
- 路由结果持久化到 `biz_route_decision`。

为什么这么做：
- 路由不应只存在于 workflow 内部，否则无法复盘和统计。

当前接口：
- `GET /routing/config`
- `GET /routes`
- `GET /routes/stats`
- `GET /tasks/{task_id}/routes`

## 6. 治理与排障设计

### 6.1 鉴权与 RBAC

- 统一鉴权支持 `API Key / Bearer`
- 最小 RBAC 支持角色、权限、主体角色分配
- 高风险接口要求显式权限

### 6.2 trace 与告警

- 每次请求生成 `trace_id`
- trace 可关联：
  - session
  - turn
  - task
  - alert
- 关键恢复事件写入告警表

### 6.3 恢复治理

- LLM 重试
- 工具重试
- LLM 熔断
- 工具熔断
- LLM 降级到 `mock`
- 工具 `soft-fail`

## 8. 当前边界

- 异步任务仍是本地后台执行器，不是真正的队列平台
- RBAC 仍是最小版，不是完整权限治理
- trace / alert 目前是查询接口，不是可视化平台
- 阶段 3 的监控、成本治理和模型策略引擎尚未落地
