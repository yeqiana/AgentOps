# 组件设计说明

## 1. 文档目的

本文档用于说明 AgentOps 当前核心组件、组件职责边界、组件间协作关系和后续扩展方向。

本文档重点回答以下问题：
- 当前有哪些关键组件
- 每个组件负责什么
- 为什么这样拆
- 组件之间如何协作

## 2. 组件划分原则

当前组件划分遵循以下原则：
- 先按真实职责拆分
- 不提前制造空壳组件
- 组件边界优先服务当前阶段能力
- 组件关系可支撑阶段 2 和阶段 3 扩展

## 3. 核心组件总览

当前核心组件包括：
- CLI 组件
- API 组件
- API 中间件组件
- 对话编排组件
- 上传编排组件
- Prompt 组件
- 工作流图组件
- 工作流节点组件
- 工作流注册组件
- 工具注册组件
- 工具执行组件
- 模型调用组件
- 媒体解析组件
- 持久化组件
- 追踪组件
- 告警组件
- 认证组件
- 配置中心组件
- 角色注册组件

## 4. 组件详细说明

### 4.1 CLI 组件

#### 这是什么

CLI 组件是命令行交互入口。

#### 做什么

主要职责如下：
- 接收用户命令行输入
- 驱动持续对话循环
- 输出普通回答
- 输出流式回答
- 处理退出命令

#### 为什么这样做

原因如下：
- 为本地联调提供最低成本入口
- 为测试工作流主链提供直接交互面
- 避免开发阶段强依赖 Web 前端

对应文件：
- `app/presentation/cli.py`

### 4.2 API 组件

#### 这是什么

API 组件是 HTTP 接入入口。

#### 做什么

主要职责如下：
- 暴露业务接口
- 输出标准 JSON 响应
- 输出 SSE 流式响应
- 暴露查询型接口与治理型接口

#### 为什么这样做

原因如下：
- 提供对外统一调用入口
- 为后续控制台、SDK、前端和第三方接入提供协议基础

对应文件：
- `app/presentation/api/app.py`
- `app/presentation/api/schemas.py`

### 4.3 API 中间件组件

#### 这是什么

API 中间件组件是 Presentation 层的横切治理组件集合。

#### 做什么

主要职责如下：
- 鉴权
- 限流
- 幂等
- trace 注入

#### 为什么这样做

原因如下：
- 治理逻辑不应散落在每个接口里
- 统一中间件可以前置安全与治理规则

对应文件：
- `app/presentation/api/middleware/auth.py`
- `app/presentation/api/middleware/governance.py`
- `app/presentation/api/middleware/trace.py`

### 4.4 对话编排组件

#### 这是什么

对话编排组件是标准聊天入口的应用服务组件。

#### 做什么

主要职责如下：
- 解析本轮输入
- 恢复或创建会话上下文
- 调用工作流
- 组织标准响应与流式响应

#### 为什么这样做

原因如下：
- 保证聊天入口逻辑集中
- 让 CLI 和 API 共用相同编排能力

对应文件：
- `app/application/agent_service.py`
- `app/application/services/chat_service.py`

### 4.5 上传编排组件

#### 这是什么

上传编排组件是处理上传型多模态输入的应用服务组件。

#### 做什么

主要职责如下：
- 接收上传文件
- 调用本地存储落盘
- 识别资产类型
- 标准化为 `InputAsset`
- 可选触发工具试跑

#### 为什么这样做

原因如下：
- 上传链路和普通文本输入链路不同
- 单独编排更易扩展上传策略与安全规则

对应文件：
- `app/application/upload_service.py`
- `app/infrastructure/storage/local_upload_store.py`

### 4.6 Prompt 组件

#### 这是什么

Prompt 组件是工作流角色提示词的统一构造器。

#### 做什么

主要职责如下：
- 构建 planner Prompt
- 构建 executor Prompt
- 构建 critic Prompt
- 构建 reviewer Prompt
- 构建 arbitration Prompt
- 拼装工具结果和历史上下文

#### 为什么这样做

原因如下：
- 避免提示词散落在节点内部
- 为后续 Prompt 版本治理保留集中入口

对应文件：
- `app/application/prompt_builder.py`

### 4.7 工作流图组件

#### 这是什么

工作流图组件是 LangGraph 图定义组件。

#### 做什么

主要职责如下：
- 定义节点顺序
- 编译工作流图
- 暴露统一执行入口

#### 为什么这样做

原因如下：
- 工作流顺序不应散落在各节点内部
- 图定义单独维护便于未来切换协议和扩展节点

对应文件：
- `app/workflow/graph.py`

### 4.8 工作流节点组件

#### 这是什么

工作流节点组件是当前 Agent 主执行链的核心业务执行组件。

#### 做什么

主要职责如下：
- `tool_node`：识别资产并调用工具
- `router_node`：决定当前路由类型
- `protocol_node`：决定执行协议
- `plan_node`：生成计划
- `debate_node`：生成多角色观点
- `arbitration_node`：仲裁多角色观点
- `answer_node`：生成最终回答
- `critic_node`：批评最终回答
- `review_node`：给出最终复核结论

#### 为什么这样做

原因如下：
- 让每个步骤都可追踪、可替换、可扩展
- 为多 Agent 最小编排提供明确执行边界

对应文件：
- `app/workflow/nodes.py`

### 4.9 工作流策略组件

#### 这是什么

工作流策略组件是执行决策规则组件。

#### 做什么

主要职责如下：
- 路由选择
- 协议选择
- 角色与流程决策规则

#### 为什么这样做

原因如下：
- 策略逻辑不应直接写死在节点里
- 便于后续进一步配置化或策略中心化

对应文件：
- `app/workflow/policies.py`

### 4.10 工作流注册组件

#### 这是什么

工作流注册组件是工作流级配置与角色读取组件。

#### 做什么

主要职责如下：
- 读取运行时工作流配置
- 读取角色协议定义
- 暴露当前有效配置给工作流层

#### 为什么这样做

原因如下：
- 让工作流不直接依赖环境变量和数据库读取细节
- 为后续角色协议扩展提供统一入口

对应文件：
- `app/workflow/registry.py`

### 4.11 模型调用组件

#### 这是什么

模型调用组件是统一的大模型访问组件。

#### 做什么

主要职责如下：
- 标准调用
- 流式调用
- provider 兼容
- 重试
- 熔断
- 降级到 mock

#### 为什么这样做

原因如下：
- 所有模型调用行为应集中治理
- 模型错误恢复不应散落在业务层

对应文件：
- `app/infrastructure/llm/client.py`

### 4.12 工具注册组件

#### 这是什么

工具注册组件是系统工具能力发现与登记组件。

#### 做什么

主要职责如下：
- 注册本地工具
- 根据环境发现工具
- 生成工具描述
- 输出可用工具清单

#### 为什么这样做

原因如下：
- 工具能力需要统一入口
- 不同节点和接口应复用同一份工具注册视图

对应文件：
- `app/infrastructure/tools/registry.py`

### 4.13 工具执行组件

#### 这是什么

工具执行组件是本地工具命令实际执行组件。

#### 做什么

主要职责如下：
- 执行 OCR、ASR、FFmpeg 等工具
- 处理超时
- 处理重试
- 处理熔断
- 处理 soft-fail 降级

#### 为什么这样做

原因如下：
- 工具执行属于高失败风险区
- 必须集中治理恢复策略和执行结果结构

对应文件：
- `app/infrastructure/tools/local_runner.py`
- `app/infrastructure/tools/adapters.py`
- `app/infrastructure/tools/retry_policy.py`
- `app/infrastructure/tools/failure_recovery.py`

### 4.14 媒体解析组件

#### 这是什么

媒体解析组件是图片、音频、视频、文件真实解析组件。

#### 做什么

主要职责如下：
- 读取媒体文件
- 提取元数据
- 提取文本
- 为工作流生成标准化资产信息

#### 为什么这样做

原因如下：
- 多模态处理不应由工作流节点直接读取底层文件
- 媒体解析与业务决策应解耦

对应文件：
- `app/infrastructure/media/image_loader.py`
- `app/infrastructure/media/audio_loader.py`
- `app/infrastructure/media/video_loader.py`
- `app/infrastructure/media/file_loader.py`

### 4.15 持久化组件

#### 这是什么

持久化组件是数据库结构和仓储实现组件。

#### 做什么

主要职责如下：
- 初始化数据库结构
- 维护结构版本
- 实现会话、任务、工具结果等读写
- 提供告警、trace、配置和角色查询支持

#### 为什么这样做

原因如下：
- 所有落库逻辑应集中在基础设施层
- 应用层只依赖服务编排，不依赖 SQL 细节

对应文件：
- `app/infrastructure/persistence/database.py`
- `app/infrastructure/persistence/repositories.py`

### 4.16 追踪组件

#### 这是什么

追踪组件是 trace 记录与检索组件。

#### 做什么

主要职责如下：
- 记录请求 trace
- 关联 task 与 alert
- 提供 trace 查询

#### 为什么这样做

原因如下：
- 企业级底座必须具备可排障链路
- trace 需要和 API、中间件、恢复链路统一协作

对应文件：
- `app/infrastructure/trace/service.py`

### 4.17 告警组件

#### 这是什么

告警组件是恢复治理和异常事件持久化组件。

#### 做什么

主要职责如下：
- 记录恢复告警
- 查询告警
- 按 trace 关联告警

#### 为什么这样做

原因如下：
- 失败恢复不能只有日志
- 必须有结构化告警记录供接口和后续观测面板使用

对应文件：
- `app/infrastructure/alert/service.py`
- `app/application/services/alert_service.py`

### 4.18 认证组件

#### 这是什么

认证组件是统一鉴权实现组件。

#### 做什么

主要职责如下：
- API Key 认证
- Bearer Token 认证
- 最小 RBAC 授权
- 输出认证主体信息

#### 为什么这样做

原因如下：
- 认证实现不应混入业务编排层
- 最小授权也不应散落在每个接口内部
- 为后续权限系统扩展保留统一接入点

对应文件：
- `app/infrastructure/auth/service.py`
- `app/presentation/api/middleware/auth.py`

### 4.19 配置中心组件

#### 这是什么

配置中心组件是运行时策略读取与更新组件。

#### 做什么

主要职责如下：
- 读取运行时配置
- 更新运行时配置
- 数据库配置优先
- 环境变量兜底

#### 为什么这样做

原因如下：
- 策略不应长期写死在代码中
- 后续阶段需要可治理、可调整、可审计的配置入口

对应文件：
- `app/application/services/config_service.py`
- `app/infrastructure/persistence/repositories.py`

### 4.20 角色注册组件

#### 这是什么

角色注册组件是多 Agent 角色协议管理组件。

#### 做什么

主要职责如下：
- 维护角色定义
- 输出当前角色协议
- 支持数据库驱动角色更新

#### 为什么这样做

原因如下：
- 多 Agent 能力不能长期硬编码为固定角色
- 角色协议需要逐步走向配置化和中心化

对应文件：
- `app/application/services/workflow_role_service.py`
- `app/workflow/registry.py`

## 5. 组件交互原则

当前组件交互遵循以下规则：
- Presentation 不直接访问数据库
- Presentation 不直接执行工具
- Application 不直接写 SQL
- Workflow 不直接处理上传文件流
- Workflow 不直接读环境变量做复杂策略判断
- Infrastructure 不承载业务编排逻辑
- 配置与角色优先通过服务和注册中心读取

## 6. 当前组件边界中的预留能力

以下能力当前已经预留接口或边界，但尚未完整实现：
- 完整 RBAC 组件
- 异步任务 worker 组件
- 可视化 trace 组件
- 更细粒度策略中心组件
- 更复杂多 Agent 协调组件

## 7. 结论

当前组件设计已经具备以下特征：
- 组件边界清晰
- 核心链路完整
- 治理组件已初步成型
- 多 Agent 最小编排已进入真实执行链
- 后续阶段可以在现有组件边界上继续演进，而无需立即进行大规模目录重构
