# 设计方案

## 1. 方案目标

为项目建立一个可以长期演进的 Agent 底座，而不是单点聊天脚本。

## 2. 方案选择

### 2.1 分层方案

选择：
- 五层结构

原因：
- 比单文件脚本更可维护
- 比过度拆分更适合当前规模

### 2.2 工作流方案

选择：
- LangGraph

原因：
- 适合状态流转
- 适合后续扩展 router / review / multi-agent

### 2.3 持久化方案

选择：
- 阶段 1 使用 SQLite

原因：
- 快速落地
- 便于测试

### 2.4 工具方案

选择：
- 本地工具注册中心 + 命令适配器

原因：
- 支持 OCR / ASR / 视频后处理
- 模型能力不足时可补强

### 2.5 请求路由中台方案

选择：
- 当前阶段保留最小路由实现
- 后续阶段演进为独立请求路由中台

原因：
- 当前项目仍需保持主流程简洁
- 现阶段以资产类型驱动的最小路由更易稳定落地
- 后续需要按任务类型、权限、模型策略和多 Agent 路由统一治理

### 2.6 异步任务体系方案

选择：
- 当前阶段保留同步执行主链
- 后续阶段新增异步任务提交、排队、执行与回查体系

原因：
- 当前阶段核心目标是先打通可执行、可追踪、可排障闭环
- 长耗时任务、批处理任务和后台执行能力需要独立任务基础设施承载

### 2.7 权限控制方案

选择：
- 采用统一鉴权入口 + 资源级授权 + 工具级授权的分层方案

原因：
- 权限不应散落在工作流节点中
- 后续企业场景必须支持用户、角色、租户和工具调用审计

### 2.8 多 Agent 编排方案

选择：
- 当前保留单 Agent 默认执行图
- 后续增加 Agent Router、角色 Agent 和结果汇聚机制

原因：
- 先保证单 Agent 执行链稳定
- 多 Agent 需要建立在统一状态、任务、路由和追踪模型之上

## 3. 当前方案收益

- 稳定执行
- 可排障
- 易扩展

## 4. 后续阶段建议

- 阶段 2 做请求路由能力增强、统一鉴权、多 Agent 最小编排
- 阶段 2 为异步任务体系补齐任务模型、状态流转和执行接口预留
- 阶段 3 做异步任务平台化、可视化观测和更完整的权限治理
