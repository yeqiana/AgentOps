# AgentOps Token 治理方案 v1

## 一、背景

在 AgentOps 体系中，Token 成本是核心运行成本之一。

随着系统能力增强（多工具、多步骤、多模态、长上下文），Token 消耗将呈现以下特点：

- 非线性增长
- 隐性放大（重复上下文、重试、长文本）
- 难以直观感知（单次看不贵，总体很高）

本方案目标是：

> 在不影响能力的前提下，建立可观测、可控制、可优化的 Token 使用体系。

---

## 二、目标

### 核心目标

1. 可观测：知道 Token 花在哪
2. 可归因：知道是哪一层消耗
3. 可控制：能限制与降级
4. 可优化：逐步降低单位请求成本

---

## 三、Token 分类模型

将 Token 消耗拆为三类：

### 1. 开发 Token（低优先级）
- Codex / AI 协作开发
- 架构设计
- UI 优化
- 文档生成

### 2. 运行 Token（核心关注）
- 用户请求触发
- Agent 执行链路
- 多轮推理
- 工具调用决策

### 3. 放大 Token（高风险）
- 长文本（OCR / ASR / PDF）
- 长会话历史
- 重复上下文拼接
- 失败重试链路

---

## 四、关键治理策略

### 1. 上下文裁剪
- 限制历史轮数
- 使用摘要替代全量历史

### 2. 分层执行
- 简单请求直接回答
- 复杂请求再进入 Agent 流程

### 3. 模型分级
- 轻模型用于 route / review
- 重模型用于复杂推理

### 4. 工具结果控制
- 不回灌原始全文
- 只回灌摘要

### 5. 会话压缩
- 定期生成 summary
- 保留关键上下文

### 6. 重试控制
- 只重试失败节点
- 限制重试次数

---

## 五、监控设计

必须记录：

### 每个请求
- total_token
- input_token
- output_token

### 每个节点
- node_type
- token_usage
- latency

---

## 六、执行路线

### 第一阶段
- 记录 token
- 限制上下文

### 第二阶段
- 引入摘要机制
- session 压缩

### 第三阶段
- 成本 dashboard
- 自动降级

---

## 七、核心原则

> 低频不优化，高频必须优化  
> 原文不入模，摘要先入模  
> 能规则不用模型，能轻模型不用重模型
