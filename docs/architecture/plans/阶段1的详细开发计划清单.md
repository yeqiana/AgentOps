# 阶段 1 的详细开发计划清单

## 阶段目标

阶段 1 的目标不是继续把项目做成更复杂的聊天程序，而是把当前仓库稳定升级成“分析型 Agent 后端底座”。

这一阶段完成后，底座至少应具备：

- 统一 API 入口
- 会话、消息、资产的数据库持久化
- 最小可用的本地工具调用能力
- 基础执行上下文标识
- 能通过日志和数据库记录排查问题

## 阶段范围

阶段 1 聚焦 5 条主线：

1. API 层
2. 数据库与 Repository 层
3. 本地工具调用层
4. 执行上下文与错误模型
5. 多模态分析能力增强

不在本阶段内的内容：

- 多 Agent 辩论
- 复杂 router / review 图
- 浏览器插件正式接入
- 微信自动化正式接入
- AI 图片 / 视频生成
- 完整认证与权限体系

这些属于阶段 2 或阶段 3。

## 当前判断：前期必须打底 vs 可以后置

### 前期必须打底的底座项

这些能力要在阶段 1 前半段先落下来，否则后面功能越加越散：

1. API 骨架
2. 数据库核心表
3. Repository 抽象
4. 工具网关最小骨架
5. 执行上下文标识
6. 基础错误模型

### 可以在底座骨架完成后继续做的增强项

这些重要，但不该抢在底座前面：

1. PDF 深度解析
2. OCR / ASR / 视频抽帧
3. 任务状态查询增强
4. 后台管理字段
5. 更多资产元数据
6. 更复杂的工作流节点

## 模块拆分

### 模块 A：API 基础设施

优先级：`P0`

这是什么：

- 对外统一的后端入口层

做什么：

- 提供 HTTP API
- 承接聊天、会话、资产分析等请求

为什么现在就做：

- 没有统一 API，后面 Web、小程序、App、插件都无法复用底座能力

建议目录：

```text
app/presentation/api/
|-- __init__.py
|-- app.py
|-- routes/
|   |-- chat.py
|   |-- session.py
|   |-- asset.py
|   `-- health.py
`-- schemas/
    |-- request.py
    `-- response.py
```

交付内容：

- FastAPI 应用入口
- 健康检查接口
- 对话接口
- 资产上传 / 分析触发接口
- 会话查询接口

验收标准：

- 可以通过 HTTP 发起一轮对话
- 可以通过 HTTP 提交文件或媒体分析请求
- 会话和消息可以通过接口查询

### 模块 B：数据库与 Repository

优先级：`P0`

这是什么：

- 持久化层和仓储抽象层

做什么：

- 把用户、会话、消息、资产落库
- 为任务、记忆、工具调用留出扩展位

为什么现在就做：

- 没有持久化，就没有真正的状态底座，也谈不上稳定排障

建议目录：

```text
app/domain/repositories/
|-- user_repository.py
|-- session_repository.py
|-- message_repository.py
`-- asset_repository.py

app/infrastructure/persistence/
|-- __init__.py
|-- database.py
|-- models/
|-- repositories/
`-- migrations/
```

第一阶段必须落的表：

- `users`
- `sessions`
- `messages`
- `assets`

建议技术路线：

- 本地开发：SQLite
- 后续升级：PostgreSQL

交付内容：

- 数据库连接层
- ORM 模型
- Repository 实现
- 初始化脚本 / migration 方案

验收标准：

- 一轮对话后，会话和消息可写入数据库
- 一次图片 / 音频 / 视频 / 文件分析后，资产可写入数据库
- 能通过 session 查询完整消息和资产列表

### 模块 C：本地工具调用层

优先级：`P0`

这是什么：

- 模型之外的统一工具网关最小骨架

做什么：

- 当 GPT、豆包等模型能力不足时，调用本地工具补足分析能力

为什么现在就做：

- 这是你底座区别于普通聊天项目的关键能力，必须尽早标准化

建议目录：

```text
app/infrastructure/tools/
|-- __init__.py
|-- local_runner.py
|-- registry.py
`-- adapters/
    |-- ocr_adapter.py
    |-- asr_adapter.py
    |-- video_adapter.py
    `-- draw_adapter.py
```

最小可交付能力：

- 注册工具
- 调用本地命令或本地脚本
- 捕获 stdout / stderr
- 统一返回结构
- 失败时写日志

交付内容：

- 本地工具执行器
- 工具注册机制
- 统一结果对象

验收标准：

- 可以调用一个本地程序处理输入文件
- 工具成功 / 失败都有统一返回
- 可以把工具结果交还给应用层继续分析

### 模块 D：执行上下文与错误模型

优先级：`P0`

这是什么：

- 可追踪执行链路的最小基础设施

做什么：

- 给一次执行补齐上下文标识
- 给错误补齐分类和统一对象

为什么现在就做：

- 没有这个层，就还达不到“出问题时能查清楚”

建议交付内容：

- `session_id`
- `turn_id`
- `task_id`
- `trace_id`
- 基础错误类型：
  - 模型错误
  - 工具错误
  - 解析错误
  - 持久化错误
  - 参数错误

验收标准：

- 一次请求能串起日志上下文
- 工具调用和模型调用能关联到同一个 trace
- API 返回错误时能区分错误类别

### 模块 E：多模态分析增强

优先级：`P1`

这是什么：

- 对已有多模态解析能力的增强层

做什么：

- 补足更强的真实分析能力

为什么放在底座骨架之后：

- 这属于能力增强，不是底座最小闭环

当前已有：

- 图片真实读取
- WAV 元数据解析
- MP4/MOV、AVI 容器级元数据解析
- 文本文件与 DOCX 提取

这一阶段建议继续补：

- PDF 解析
- Excel / CSV 处理
- 图片 OCR 接入本地工具
- 音频 ASR 接入本地工具
- 视频抽帧 / 抽音轨接入本地工具

验收标准：

- PDF 至少能提取正文或可读文本摘要
- 音频至少能接本地 ASR 工具
- 视频至少能接本地抽帧 / 抽音轨工具

### 模块 F：会话与任务服务

优先级：`P1`

这是什么：

- 应用层的后端服务编排层

做什么：

- 把 API、workflow、persistence、tools 串起来

为什么放在这一层：

- 避免 API 直接操作 workflow 和 persistence 细节

建议目录：

```text
app/application/services/
|-- agent_service.py
|-- session_service.py
`-- task_service.py
```

交付内容：

- 创建会话
- 获取会话历史
- 创建分析任务
- 查询任务结果

验收标准：

- API 层不直接操作 workflow 和 persistence 细节
- 业务动作由 service 层统一编排

### 模块 G：测试体系升级

优先级：`P1`

这是什么：

- 阶段 1 的稳定性保障层

做什么：

- 给 API、数据库、工具调用和多模态扩展补自动化验证

为什么放在阶段 1：

- 没有测试，底座一扩就容易回归

建议目录：

```text
tests/
|-- unit/
|-- integration/
`-- e2e/
```

本阶段新增测试重点：

- API 冒烟测试
- Repository 测试
- 本地工具调用测试
- 文件 / 音视频扩展解析测试

验收标准：

- 关键链路都具备自动化验证
- 改动后能快速发现回归

## 实施顺序

推荐按下面顺序做，不要打乱。

### 第 1 步：落 API 骨架

原因：

- 没有统一 API，后面 Web、小程序、App 都没法共用底座能力

输出：

- `presentation/api/`
- 基础路由

### 第 2 步：落数据库基础表

原因：

- API 一旦接入，就需要会话、消息、资产持久化

输出：

- `users`
- `sessions`
- `messages`
- `assets`

### 第 3 步：落 Repository 抽象

原因：

- 让 API 层不直接写数据库，也不直接耦合底层存储实现

输出：

- `domain/repositories`
- `infrastructure/persistence/repositories`

### 第 4 步：落执行上下文与错误模型

原因：

- 这是“稳定执行、出问题可查”的前置条件

输出：

- `trace_id / session_id / turn_id / task_id`
- 错误对象和错误分类

### 第 5 步：落工具网关最小骨架

原因：

- 这是底座与普通聊天项目拉开差距的关键能力

输出：

- `infrastructure/tools`
- 工具注册机制
- 本地工具执行器

### 第 6 步：落 Service 编排层

原因：

- 让 API 只做入口，让业务编排回到应用层

输出：

- `application/services`

### 第 7 步：增强多模态分析

原因：

- 底座骨架有了，再补强能力更稳

输出：

- PDF / OCR / ASR / 视频抽帧接入点

### 第 8 步：补 API / DB / Tools 测试

原因：

- 不补自动化测试，后续一扩结构就容易回归

### 里程碑 1：统一入口可用

交付：

- API 可启动
- 聊天接口可用
- 健康检查可用

### 里程碑 2：状态可持久化

交付：

- 会话、消息、资产落库
- API 可查询历史

### 里程碑 3：执行可追踪

交付：

- 请求具有 `trace_id / session_id / turn_id / task_id`
- 错误有基础分类
- 日志可以串起一次执行链路

### 里程碑 4：工具补能力可用

交付：

- 本地工具调用可用
- 模型不足时可走工具补充

### 里程碑 5：多模态增强落地

交付：

- PDF / OCR / ASR / 视频处理至少落一部分

## 风险点

### 1. 范围蔓延

风险：

- 很容易一边做 API，一边又想做多 Agent，又一边想做生成能力

控制方式：

- 阶段 1 严格只做“分析型底座”

### 2. 工具系统做成硬编码

风险：

- 本地工具调用如果直接写死在某个 service 里，后面会很难扩展

控制方式：

- 必须先做 `registry + runner`

### 3. 数据库模型和领域模型耦合过重

风险：

- 后面换数据库会很痛苦

控制方式：

- `domain/repositories` 定义接口
- `infrastructure/persistence` 实现接口

### 4. 没有统一 trace 标识

风险：

- 后面日志、数据库、工具调用、API 返回会完全对不上

控制方式：

- 在阶段 1 早期就补齐 `trace_id / session_id / turn_id / task_id`

## 一句话结论

阶段 1 不应该继续围绕“聊天”打转，而应该明确建设成：

- 有 API 的后端底座
- 有数据库的状态底座
- 有工具网关的分析底座
- 有执行上下文和错误模型的可排障底座

这一步打稳之后，阶段 2 的多 Agent 和阶段 3 的平台化才站得住。
