# 当前底座成熟度评估

## 评估标准

这里的判断标准不是“能不能跑起来”，而是是否已经接近下面这句话：

> 给我一个模型 + 一组工具 + 一段任务指令，我能稳定执行，并且出问题时能查清楚。

这句话拆开后，底座至少要满足 4 类能力：

1. 模型接入能力
2. 工具调用能力
3. 稳定执行能力
4. 可追踪、可排障能力

## 当前已具备

### 1. `agent-runtime` 基础版

当前已经具备：

- 五层结构
- CLI 持续对话
- LangGraph 基础工作流
- 统一状态对象
- 基础 Prompt 编排

结论：

- 已经具备 Agent Runtime 雏形
- 但还不是面向多入口、多任务、多用户的完整运行时

### 2. `model-adapter` 基础版

当前已经具备：

- 多 Provider 接入
- OpenAI 兼容协议调用
- mock provider
- 图片多模态输入

结论：

- 模型接入层已经成型
- 但还没有正式的模型路由策略

### 3. `multimodal-input` 基础版

当前已经具备：

- 文本、图片、音频、视频、文件的统一输入抽象
- 本地路径、URL、Base64、对象存储 URI 等入口
- 基础元数据解析

结论：

- 已经不是纯文本聊天项目
- 但多模态能力更多还停留在“读取与基础解析”，还不是完整分析引擎

### 4. `logging` 初级版

当前已经具备：

- 日志开关
- 日志级别
- 对话上下文打印
- 基础异常提示

结论：

- 已经有可用日志
- 但还不是真正的 trace service

## 当前缺失

### 1. `tool-gateway`

当前缺失：

- 工具注册中心
- 工具调用协议
- 工具结果统一结构
- 工具超时、重试、失败分类
- 模型与工具协同的标准入口

影响：

- 现在还不能说“给一组工具就能稳定执行”

### 2. `execution-engine`

当前缺失：

- 任务状态机
- 同步 / 异步任务统一抽象
- 长任务执行模型
- 执行记录持久化

影响：

- 现在更像“会话驱动的 graph 调用”
- 还不是完整执行引擎

### 3. `trace-service`

当前缺失：

- `trace_id`
- `session_id`
- `turn_id`
- `task_id`
- `tool_call_id`
- 可按执行链路查询的持久化记录

影响：

- 现在能看日志
- 但还做不到稳定复盘一次任务执行链路

### 4. `persistence`

当前缺失：

- 用户、会话、消息、资产正式落库
- 任务和工具调用落库
- 查询接口与仓储层

影响：

- 现在问题排查仍然主要依赖本地日志
- 还不是后端系统级别的可追踪能力

### 5. `api-runtime`

当前缺失：

- 正式 HTTP API
- 后续 WebSocket / SSE
- 对外统一调用入口

影响：

- 当前底座更像本地单机原型
- 还不是可供 Web、小程序、App、插件统一复用的后端底座

### 6. `auth-service`

当前缺失：

- 用户认证
- 权限控制
- 多租户隔离

影响：

- 还不具备平台级后端能力

## 当前成熟度判断

如果按平台底座标准拆分，当前状态可以这样判断：

- `agent-runtime`：基础版，已具备
- `model-adapter`：基础版，已具备
- `multimodal-input`：基础版，已具备
- `tool-gateway`：未正式落地
- `execution-engine`：仅有轻量雏形
- `trace-service`：只有日志版
- `persistence`：未正式落地
- `api-runtime`：未正式落地
- `auth-service`：未开始
- `model-router`：未正式落地

## 哪些需要前期就打底

这些能力必须在阶段 1 前半段就落下来，否则后面越做越散：

### 1. API 入口骨架

原因：

- 没有统一入口，Web、小程序、App、插件都没法复用底座能力

### 2. Repository 抽象和数据库核心表

原因：

- 不先把会话、消息、资产落库，后面 trace、任务、工具调用都会悬空

### 3. 工具网关最小骨架

原因：

- 你“模型不够时调用本地软件”的核心能力，必须尽早形成标准接口

### 4. 统一执行上下文标识

至少先补：

- `session_id`
- `turn_id`
- `task_id`
- `trace_id`

原因：

- 不先打这个底，后面日志、数据库、API、任务系统都难以对齐

### 5. 基础错误模型

至少先区分：

- 模型错误
- 工具错误
- 解析错误
- 持久化错误
- 参数错误

原因：

- 否则“能查清楚”只是口号，实际会退化成一堆字符串日志

## 哪些可以在底座成型后继续做

这些能力重要，但不该抢在底座骨架前面：

### 1. 更强的多模态增强

例如：

- PDF 深度解析
- OCR 接入
- ASR 接入
- 视频抽帧
- 音轨分离

原因：

- 这些属于能力增强，不是底座结构本身

### 2. 多 Agent 辩论与复杂工作流

例如：

- `router_node`
- `review_node`
- `debate_graph`

原因：

- 这是阶段 2 的工作流复杂化，不是阶段 1 的底座必需品

### 3. 浏览器插件、微信自动化、语音助手、智能设备

原因：

- 这些是接入场景，不是底座最小闭环

### 4. AI 图片 / 视频生成

原因：

- 这是平台化后期能力，复杂度和资源成本都高

### 5. 认证与权限体系

原因：

- 单机原型演进到多人平台时再补更合理

## 阶段 1 的正确达标线

阶段 1 不应该宣称“已经是完整 Agent 平台”，而应该把达标线设成：

### 必须达到

- 有统一 API 入口
- 有数据库核心表和仓储层
- 有最小工具网关
- 有执行上下文标识
- 有基础错误模型
- 有会话、消息、资产的可查询能力
- 有日志与数据库结合的可排障基础

### 可以暂缓

- 多 Agent
- 辩论
- 浏览器插件正式接入
- 微信自动化
- 智能家具
- AI 图片生成
- AI 视频生成

## 一句话结论

当前底座已经具备 Agent Runtime 原型，但还没有达到“给模型 + 给工具 + 给任务即可稳定执行并且可排障”的工程化标准。

阶段 1 的重点不是继续堆能力，而是补齐：

- API
- Persistence
- Tool Gateway
- Execution Context
- Error Model

这 5 个底座支点。
