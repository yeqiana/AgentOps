# 适合这个项目下一阶段落地的精简目录方案

## 目标

这份方案面向“下一阶段就要落地”的 Agent 底座演进，不追求一步到位的大而全，而是优先保证：

- 当前项目能平滑迁移
- 目录结构更清晰
- 后续接 API、数据库、工具节点时不用推翻重来
- 现有多模态分析能力可以继续扩展
- 为多 Agent、浏览器插件、自动化应用预留演进空间

## 这套底座未来要承接什么

把你的长期想法收敛后，下一阶段应该围绕 4 条主线建设：

### 1. 多模态分析底座

负责：

- 文本分析
- 图片分析
- 音频分析
- 视频分析
- 文件分析

设计要求：

- 模型能直接做时，优先走模型
- 模型做不了时，允许调用本地软件、本地工具链或外部程序

### 2. 平台后端底座

负责：

- 用户、会话、消息、资产管理
- API 服务
- 后台任务记录
- 常规后端能力

### 3. 多 Agent 编排底座

负责：

- 多模型协同
- 多角色对话
- AI 辩论
- review / router / tool 流程

### 4. 外部应用接入底座

负责：

- 浏览器插件
- Web 页面
- 小程序
- App
- 自动化工具
- 语音助手
- 智能家居

## 设计原则

- 保留现有五层结构
- 先解决高频扩展点，不做过度设计
- 先分析型，再编排型，再平台型
- 让目录先服务于后续演进，而不是只服务当前 CLI

## 这个结构为什么适合下一阶段

### 1. 改动量可控

你已经有：

- `presentation`
- `application`
- `domain`
- `workflow`
- `infrastructure`

所以现在不是推翻重来，而是正式收口：

- `llm/`
- `media/`
- `tests/`
- `docs/`
- `api/` 预留
- `persistence/` 预留
- `tools/` 预留

### 2. 把未来变化最大的地方先隔离出来

最值得先隔离的是：

- `infrastructure/llm/`
- `infrastructure/media/`
- `infrastructure/tools/`

因为这三个区域将来最容易继续扩张：

- 新增 Provider
- 新增 OCR / ASR / 视频抽帧
- 新增本地软件调用
- 新增浏览器自动化 / 微信自动化 / 设备控制

### 3. 仍然保持简单

现在还不建议立刻拆成非常细的：

- `use_cases/`
- `repositories/`
- `policies/`
- `container/`

否则目录复杂度会先超过项目复杂度。

## 每个预留位未来做什么

### `presentation/api/`

为后续 FastAPI / WebSocket 预留。

它会承接：

- Web 端聊天
- 小程序接口
- App 接口
- 浏览器插件回传

### `infrastructure/tools/`

为“模型不够时，调用本地软件、本地程序、外部工具链”预留。

它会承接：

- 本地 OCR
- 本地 ASR
- 视频工具
- 图像绘制工具
- 自动化脚本

### `infrastructure/persistence/`

为数据库和 repository 实现预留。

它会承接：

- 用户
- 会话
- 消息
- 资产
- 任务
- 记忆

### `infrastructure/storage/`

为对象存储和资源落地预留。

它会承接：

- 本地文件系统
- S3
- OSS
- MinIO

## 下一阶段落地顺序

### 第一步

继续稳定 `llm/`、`media/`、`tools/` 三类底层能力。

目标：

- 多模态分析能力更稳
- 模型与本地工具协同的入口清晰

### 第二步

补 `presentation/api/`。

目标：

- Web / 小程序 / App / 插件统一走后端接口

### 第三步

补 `infrastructure/persistence/`。

目标：

- 支持用户、会话、消息、资产落库

### 第四步

再考虑更复杂的 workflow。

目标：

- 多 Agent 辩论
- 多模型协同
- 工具调用节点

## 适用阶段

适合：

- 当前项目已经能跑
- 接下来要接 API、数据库、工具能力
- 想在不推翻现有代码的前提下长期演进

不适合：

- 一开始就多团队并行大规模开发
- 一开始就要完整 SaaS 平台能力

## 一句话结论

下一阶段最合理的做法不是大重构，而是：

- 保留五层结构
- 正式分包 `llm` 和 `media`
- 预留 `tools / api / persistence / storage`
- 先把分析型底座做扎实，再继续往多 Agent 和平台化演进
