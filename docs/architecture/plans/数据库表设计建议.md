# 数据库表设计建议

## 目标

这份设计面向 Agent 底座常见的持久化需求，重点覆盖：

- 用户
- 会话
- 消息
- 输入资产
- 任务执行
- 长期记忆
- 工具调用
- 多 Agent 协作
- 生成任务

它适合后续接：

- Web
- 小程序
- App
- 浏览器插件
- 管理后台
- API 服务

## 这套数据库要服务什么目标

数据库设计不只是为了“存聊天记录”，而是为了支撑你未来这几类能力：

### 1. 平台后端能力

需要：

- 用户表
- 会话表
- 消息表
- 资产表

### 2. 多模态分析能力

需要：

- 资产表
- 附件表
- 元数据字段

### 3. 多 Agent 协作能力

需要：

- 任务表
- 角色表
- 辩论轮次表
- 工具调用表

### 4. 生成式任务能力

需要：

- 生成任务表
- 输出产物表
- 存储映射表

## 设计原则

- 会话和消息分离
- 输入资产单独建表
- 任务执行单独建表
- 记忆单独建表
- 工具调用单独建表
- 生成结果单独建表
- 尽量保留可审计性

## 推荐表关系

```text
users 1 --- n sessions
sessions 1 --- n messages
messages 1 --- n assets
sessions 1 --- n task_runs
task_runs 1 --- n tool_calls
users 1 --- n memories
sessions 1 --- n memories
task_runs 1 --- n debate_rounds
agent_roles 1 --- n debate_rounds
task_runs 1 --- n generation_tasks
assets 1 --- 0..1 attachments
```

## 分阶段建表建议

### 第一阶段：分析型底座

先建：

- `users`
- `sessions`
- `messages`
- `assets`

这已经够支撑：

- 多轮对话
- 多用户
- 多模态资产
- 基础历史查询

### 第二阶段：编排型底座

再建：

- `task_runs`
- `tool_calls`
- `memories`

这时就能支撑：

- 工具调用
- 工作流状态跟踪
- 长期记忆

### 第三阶段：平台型 / 生成型底座

最后补：

- `agent_roles`
- `debate_rounds`
- `generation_tasks`
- `attachments`

这时就能支撑：

- 多 AI 辩论
- 生成任务
- 大文件存储映射

## 数据库选型建议

### 起步阶段

推荐：

- SQLite

### 正式服务阶段

推荐：

- PostgreSQL

原因：

- JSON 能力强
- 索引能力强
- 事务和扩展能力适合 Agent 平台

### 缓存

后续推荐：

- Redis

适合：

- 会话缓存
- 任务状态缓存
- 限流
- Prompt / 模型结果缓存

## 一句话结论

如果这个项目的目标是 Agent 平台底座，那么数据库设计至少要支撑 4 类对象：

- 平台后端对象：用户、会话、消息
- 多模态对象：资产、附件
- 编排对象：任务、工具调用、记忆
- 平台扩展对象：角色、辩论轮次、生成任务

先把第一阶段建稳，再逐步往后扩展，才是最稳的路线。
