# 企业级完整版目录方案

## 目标

这份方案面向“长期演进”和“企业级协作”的 Agent 底座。

它适用于以下场景：

- 多终端接入：CLI、Web、App、小程序、浏览器插件
- 多 Provider 模型接入
- 多模态分析
- 多 Agent 协作与辩论
- 数据库持久化
- 对象存储
- 异步任务
- 自动化工具接入
- AI 图片 / AI 视频生成

## 未来能力主线

企业级底座不应该只是一套目录，而应该能承接 4 类核心能力：

### 1. 分析型底座

负责：

- 文本分析
- 图片分析
- 音频分析
- 视频分析
- 文件分析

### 2. 编排型底座

负责：

- 多模型协同
- 多角色对话
- AI 辩论
- planner / router / review / tool 工作流

### 3. 平台型底座

负责：

- 用户体系
- 会话管理
- 任务中心
- 资产管理
- API 服务
- 管理后台
- 统一鉴权
- trace service
- 可视化观测
- 配置中心
- 版本治理
- 安全策略
- 限流与配额
- 成本治理

### 4. 落地型底座

负责：

- 浏览器插件
- 微信自动化
- 语音助手
- 智能家具
- AI 图片生成
- AI 视频生成

## 设计原则

- 表现层可扩展
- 领域层稳定
- 工作流层独立
- 基础设施可替换
- 数据持久化和外部依赖全部下沉
- 支持多人并行协作

## 目录如何承接你的长期目标

### 想做本地软件调用

落点：

- `infrastructure/tools/`

用途：

- 模型不足时调用本地工具读取、分析、绘制

### 想做多 Agent 辩论

落点：

- `application/use_cases/multi_agent_debate.py`
- `workflow/graphs/debate_graph.py`
- `workflow/nodes/debate_node.py`

### 想做浏览器插件、微信、语音助手、智能家具

落点：

- `presentation/gateway/`
- `infrastructure/tools/automation_tools.py`

### 想做 AI 图片 / AI 视频生成

落点：

- `application/use_cases/generate_image.py`
- `application/use_cases/generate_video.py`
- `workflow/graphs/generation_graph.py`
- `infrastructure/tools/generation_tools.py`

### 想做数据库、后台、任务中心

落点：

- `domain/repositories/`
- `infrastructure/persistence/`
- `application/services/task_service.py`

### 想做统一鉴权

落点：

- `presentation/api/middleware/auth.py`
- `infrastructure/auth/`

用途：

- 提供 API Key / Bearer Token 认证
- 提供用户身份识别和入口级权限校验

### 想做失败重试与恢复

落点：

- `infrastructure/tools/retry_policy.py`
- `infrastructure/tools/failure_recovery.py`
- `workflow/policies/`

用途：

- 统一模型调用和工具调用的重试、超时、退避与失败恢复策略

### 想做 trace service 和可视化 trace

落点：

- `infrastructure/trace/`
- `presentation/api/routes/task.py`
- 后续管理后台或 trace dashboard

用途：

- 存储和检索执行链路
- 为任务排障、工具排障和可视化 trace 页面提供底层数据

### 想做配置、版本和策略治理

落点：

- `infrastructure/config/`
- `application/prompts/versions/`
- `domain/services/guardrail_policy.py`
- `presentation/api/middleware/rate_limit.py`
- `presentation/api/middleware/idempotency.py`

用途：

- 管理环境配置、Provider 配置、Prompt 版本、接口版本和策略控制
- 避免多模型、多 Prompt、多入口并行扩展时失控

### 想做成本、限流和资源配额

落点：

- `application/services/cost_service.py`
- `infrastructure/governance/`
- `infrastructure/observability/`

用途：

- 做模型成本统计、并发上限控制、配额限制和告警
- 保证底座在企业场景下可持续运行

## 适用场景

适合：

- 计划做 SaaS Agent 平台
- 要接管理后台
- 要接 Web / App / 小程序 / 插件
- 要做多模态分析和多 Agent 协作
- 要做生成式工作流

## 一句话结论

企业级完整版方案的价值不在于目录多，而在于它能同时承接：

- 分析型底座
- 编排型底座
- 平台型底座
- 落地型底座

如果你的目标是长期平台化，这个结构是合理上限。
