# 前端底座与权限契约冻结说明

## 1. 背景

当前 Trace Console 前端已完成最小认证与权限骨架，包括开发凭证登录、`/auth/me` 接入、HTTP 鉴权注入、登录态守卫、菜单权限过滤与路由权限判断。

为避免后续页面重构过程中反复改动认证、权限、菜单与路由守卫基础设施，现将前端底座与权限契约冻结为后续开发基线。

本说明用于约束后续 Trace Console 页面重构、状态页收口、权限体验增强与正式认证体系接入前的开发边界。

## 2. 冻结范围

### 2.1 认证骨架

以下认证能力作为当前前端底座冻结：

- `AuthProvider`
- `useAuth`
- `authApi`
- `authStorage`
- `AuthProfile`

`AuthProfile` 的前端内部标准字段冻结为：

```ts
{
  subject,
  roles,
  permissions,
  authType
}
```

后续页面和组件不得直接依赖后端 `/auth/me` 原始字段，例如 `auth_subject`、`auth_type`。

### 2.2 本地凭证管理

当前开发凭证统一通过 `authStorage.ts` 管理。

冻结约束：

- 当前存储方式为 `localStorage`。
- key 由 `authStorage.ts` 内部集中定义。
- 除 `authStorage.ts` 外，其他业务代码不得直接读写 `localStorage` 或 `sessionStorage` 中的认证凭证。

### 2.3 HTTP 鉴权注入

当前 `getJson` 会在存在开发凭证时自动注入：

```http
X-API-Key: <credential>
```

冻结约束：

- `getJson` 调用签名保持不变。
- 业务 API 调用方不得单独拼接认证请求头。
- 后续如需支持 Bearer token、cookie 或 SSO，应单独开线程设计，不在页面重构中顺手改造。

### 2.4 登录入口

当前 `/login` 使用开发凭证登录。

冻结约束：

- 登录页成功后优先按 `location.state?.from` 回跳。
- 无来源地址时默认进入 `/console/observability`。
- 当前不引入正式用户名密码登录、refresh token、cookie session 或 SSO。

### 2.5 路由守卫

当前 `RequireAuth` 负责控制台区域的登录态守卫与最小路由权限判断。

冻结约束：

- 未登录访问控制台路由时跳转 `/login`。
- 已登录后再根据 `ROUTE_PERMISSION_MAP` 做路由核心权限判断。
- 未命中权限映射的路径默认允许访问。
- 无权限时返回最小占位；正式 403 页面后续单独实现。

### 2.6 权限工具

以下权限工具作为冻结基础能力：

- `PERMISSIONS`
- `ROUTE_PERMISSION_MAP`
- `usePermission`

`usePermission` 当前提供：

```ts
permissions
hasPermission(code)
hasAnyPermission(codes)
hasAllPermissions(codes)
```

冻结约束：

- `"*"` 作为超级权限。
- 不基于角色名做前端判断。
- 不在页面组件中散落 `profile.permissions.includes(...)`。
- 页面、菜单、路由如需判断权限，应优先使用 `usePermission`。

### 2.7 菜单权限

当前菜单项支持：

```ts
permissionCodes?: string[]
```

Sidebar 使用 `hasAnyPermission(permissionCodes)` 过滤菜单项。

冻结约束：

- 没有 `permissionCodes` 的菜单项默认可见。
- 配置了 `permissionCodes` 的菜单项按 `hasAnyPermission` 判断。
- section 过滤后为空时不渲染。
- 当前不引入动态菜单或后端菜单下发。

### 2.8 路由权限

当前核心路由权限冻结如下：

```ts
/console/observability -> task.read
/console/traces -> trace.read
/console/tasks -> task.read
```

冻结约束：

- 路由权限使用单核心 permission。
- `RequireAuth` 使用 `hasPermission(permission)` 判断。
- 同一路由应保持同一核心权限。
- 当前 `Alert Center` 指向 `/console/observability`，因此使用 `task.read`，不使用 `alert.read`。

## 3. 禁止随意改动项

后续页面重构中禁止随意改动以下内容：

- 不随意改 `AuthProfile` 字段名。
- 不绕过 `authStorage` 直接读写浏览器存储。
- 不绕过 `getJson` 或统一 HTTP 客户端单独拼接认证头。
- 不在页面组件中直接写 `profile.permissions.includes(...)`。
- 不在前端组件中写角色名判断，例如 `role === "admin"`。
- 不随意改变 `ROUTE_PERMISSION_MAP` 的核心权限口径。
- 不在页面结构未稳定前做 panel-level 权限。
- 不在页面结构未稳定前做按钮级权限。
- 不在页面重构线程中顺手引入 SSO、token、cookie 或动态菜单。

## 4. 当前明确不做

以下能力不属于当前冻结基线，后续如需实现必须单独开线程：

- panel-level 权限。
- 按钮级权限。
- 完整 403 页面。
- 正式用户名密码登录。
- refresh token。
- httpOnly cookie session。
- OAuth2 / OIDC / CAS / SAML / SSO。
- 动态菜单。
- 后端菜单下发。
- ABAC。
- 租户、组织、空间、资源级权限。

## 5. 后续复用要求

后续页面重构必须优先复用以下底座能力：

- `useAuth`
- `usePermission`
- `PERMISSIONS`
- `ROUTE_PERMISSION_MAP`
- `permissionCodes`
- `RequireAuth`
- `getJson`
- `authStorage`

新增页面时，应先明确：

- 页面是否需要登录态守卫。
- 页面对应的核心 permission。
- 是否需要新增 `PERMISSIONS` 常量。
- 是否需要补充 `ROUTE_PERMISSION_MAP`。
- 菜单入口是否需要配置 `permissionCodes`。

## 6. 安全边界说明

前端权限仅用于体验控制，包括菜单隐藏、路由阻断和减少误入无权限页面。

真实安全边界仍然后端 `require_permission`。

任何涉及数据读取、写入、执行、配置变更的能力，都必须以后端权限校验为准。前端隐藏菜单或拦截路由不能替代后端鉴权。

## 7. 下一阶段建议

下一阶段建议优先进入页面重构与状态体验收口，而不是继续扩展权限体系。

推荐顺序：

1. Trace Console 页面结构重构与状态展示收口。
2. 正式 403 页面或统一无权限状态组件。
3. 页面结构稳定后再做 panel-level 权限。
4. 最后再评估正式认证体系，包括 token、cookie、refresh token 与 SSO。
